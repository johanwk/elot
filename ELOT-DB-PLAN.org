#+title: Implementation Plan: Global Slurp & Label-Display for ELOT
#+startup: overview

* Overview

This plan describes the addition of a *global, SQLite-backed label store* to
ELOT, complementing the existing buffer-local =elot-slurp= mechanism. It
enables label-display in both ELOT =.org= files and in foreign file types
(Turtle, CSV/TSV, SPARQL queries, JSON, etc.), with per-project source
selection and cross-editor portability (Emacs now, VS Code / CLI later).

This document is the authoritative reference for the feature. It is designed
to be:
- *Self-contained* - an LLM or human contributor can pick it up without
  needing the original design conversation.
- *Incrementally updatable* - each step has a status field; a decisions log
  and changelog at the bottom capture ongoing refinements.

* Use Cases

The plan serves two distinct primary users. Keeping both visible
during implementation prevents design drift toward the first at the
expense of the second.

** UC1: ELOT author working in Emacs

An ontology author edits =.org= files under the ELOT conventions,
uses label-display in the current buffer, and benefits from global
slurp when cross-referencing identifiers defined in sibling files,
CSV code lists, Turtle imports, or SPARQL queries against local
=.ttl= files or remote endpoints (e.g. Wikidata). This is the
historical ELOT audience and the scenario the existing codebase was
written for.

** UC2: Code- and data-browsing developer (cross-editor)

A developer works with source files and data files in which RDF
identifiers appear as opaque strings - typically long GUIDs or
CURIEs embedded in Python, Java, JSON, JSON-LD, Turtle, SPARQL
results, or Jupyter notebooks. Without support, the file is
effectively unreadable; reviewers and maintainers cannot tell what
any given identifier refers to without leaving the editor.

The concrete motivating scenario: a colleague using =rdflib= in
Jupyter notebooks inside VS Code, reading and writing code that
manipulates RDF terms whose URIs convey nothing to the human reader.
Inline labels rendered beside each identifier make the code
readable, reviewable, and maintainable.

Supporting UC2 drives several decisions already captured in this
plan:
- The =elot-db-*= API is the single Emacs-specific seam; parsers
  produce a language-neutral slurp shape; =schema.sql= is shared
  verbatim with the VS Code port. A VS Code extension reading the
  =.sqlite= written by Emacs is a first-class path, not an
  afterthought (see /Portability Boundary/).
- Source registration is independent of the consumer's file type:
  a =.py= / =.ipynb= / =.java= / =.ttl= / =.rq= buffer all ask the
  same question ("given this token, what's its label?") against the
  same DB.
- =.dir-locals.el= (Emacs) and =.vscode/settings.json= (VS Code)
  both configure =elot-active-label-sources= for the same
  workspace, so the two editors show the same labels from the same
  ontologies.
- A CLI tool (Step 2.5) is the escape hatch for developers who use
  neither Emacs nor the VS Code extension directly (CI, Docker,
  server-side kernels, shared-cache population).

Known execution risks specific to UC2, to be addressed in the
relevant steps rather than here:
1. *Identifier recognition in code/data files.* CURIEs and full
   IRIs appear inside string literals, dict keys, URL fragments,
   and f-strings. V1 should start with a conservative regex (full
   IRI =<...>= and =prefix:localname= where =prefix= is a registered
   prefix) and loosen per language as needed.
2. *rdflib-flavoured identifiers.* =URIRef("http://...")= and literal
   CURIEs are in scope for v1; namespace-object dotted access
   (=EX.foo= where =EX = Namespace(...)=) requires light static
   analysis and is a v2 concern.
3. *Jupyter notebooks.* =.ipynb= is JSON and VS Code's notebook API
   exposes cells as independent editors; decoration providers must
   handle cells, not just flat text documents. Step 2.4 carries
   this as an explicit line item.
4. *UX difference between editors.* Emacs uses =display= text
   properties (in-place replacement); VS Code uses
   =TextEditor.setDecorations= with =before=/=after= content
   (adjacent annotation). For source code this is arguably
   preferable - the original identifier remains visible - but it
   is a conscious design difference to flag in user-facing docs.
5. *Populating the DB without Emacs.* Step 2.5 (CLI) is
   load-bearing for UC2: it is how a non-Emacs user refreshes the
   cache. Its status is effectively *recommended*, not "optional",
   notwithstanding the label in the milestone table.
6. *Prefix / namespace management.* A Python file's =ex:Foo= cannot
   be resolved unless something tells the system that =ex:= expands
   to the same URI the DB was populated with. Prefix-map handling
   deserves an explicit home in Stage 1 (candidate: a new Step
   1.3.5 or an addition to Step 1.5) and is called out here so it
   is not forgotten.

** UC3: Generic identifier-label mapping (any project)

A developer works on a project that has nothing to do with
ontologies but is nonetheless littered with opaque identifiers:
employee codes (=EMP-12345=) in HR tooling, product SKUs
(=ACME-SKU-9=) in inventory code, ticket IDs (=PROJ-4711=) in
commit messages and source comments, standards part numbers,
short git SHAs paired with commit subjects, database primary keys
alongside display-name lookup tables. Without support, the file
reads as a wall of meaningless tokens; with a label mapping, each
token is annotated in place with its human-readable meaning.

The value proposition is identical to UC2 - opaque tokens become
readable where they are written - but no RDF / ontology vocabulary
is involved anywhere in the pipeline. The mapping source is a
flat CSV / TSV / JSON file (=employees.csv=, =skus.json=,
=ticket-subjects.tsv=) and the consumer buffer is a =.py=, =.java=,
=.md=, =.sh=, or =.org= file containing those raw identifiers.

The architecture already supports this case with minimal
additional effort:
- The =(id, label, plist)= slurp shape is identifier-agnostic.
- The DB schema stores opaque strings; =entities.kind= defaults to
  ='unknown'=, no RDF assumption.
- The two-pass lookup's *first pass* is a literal =id= match;
  sources without any prefix/URI structure are served entirely by
  that pass, and the prefix machinery of Step 1.1.1 is simply
  never consulted.
- Active-source selection and priority via =.dir-locals.el= is
  project-generic.
- The font-lock regex for non-ELOT buffers (Step 1.7) is built
  from the literal IDs present in the active sources, so the
  scanner does not need a language- or vocabulary-specific
  tokeniser to hit =EMP-12345= in Python source.

UC3 is treated as a peer of UC1/UC2 throughout this plan: every
subsequent step must stay honest about users who have no
ontologies, no URIs, and no CURIEs, only flat =id -> label= maps
and source files referring to those ids.

* Non-goals

To keep scope honest and forestall drift into adjacent problem spaces,
this subsystem is explicitly *not* the following:

- *Not a triplestore.* The DB stores flattened =(id, label, source,
  data_source)= rows plus EAV-style =(prop, value)= attributes. It has
  no notion of graphs, named graphs, datatypes, inference, or SPARQL
  evaluation. If those are needed, use Jena/Fuseki, Oxigraph, rdflib,
  or similar - do not grow this schema toward them.
- *Not authoritative.* All contents are derived from primary sources
  (Org files, CSV/TSV, Turtle, SPARQL =.rq= queries). The DB is a
  cache; primary sources remain the source of truth.
- *Not persistent in the data-preservation sense.* The DB is
  rebuildable from its sources at any time. Losing =elot-cache.sqlite=
  is an inconvenience, not a data loss. Users and tools may delete it
  freely.
- *Not a query language target.* Consumers access data through the
  =elot-db-*= Elisp API (and the equivalent TypeScript API in the VS
  Code port), not by writing SQL or SPARQL against it. The schema is
  an implementation detail of that API.
- *Not a shared / multi-user store.* One user, one cache, local disk.
  Cross-editor sharing (Emacs <-> VS Code on the same machine) is a
  deferred stretch goal; cross-machine or multi-user sharing is out of
  scope entirely.
- *Not authoritative, not shared, not a triplestore - but
  deliberately general-purpose.* Although the subsystem grew out of
  ELOT, it is *not* restricted to ontology work: any project with an
  =id -> label= mapping (CSV/TSV/JSON) and source files that
  reference those ids can use it (see UC3). What it is *not* is a
  multi-user registry, a cross-machine service, or a system of
  record; it remains a single-user, single-machine, rebuildable
  local cache. ELOT is simply the first consumer.

If a future requirement pushes against any of these boundaries, that
is a signal to reconsider the architecture (and likely to adopt an
existing tool) rather than to extend this cache.

* Background & Current State

** The existing =elot-slurp= mechanism

When an ELOT =.org= buffer is parsed, a buffer-local variable =elot-slurp= is
populated with a list of entries in the shape:

#+begin_example
(ID LABEL PLIST)
#+end_example

where =PLIST= is a flat list =("prop1" "val1" "prop2" "val2" ...)=.
Example entry:

#+begin_example
("obo:BFO_0000001" "entity"
 ("rdf:type" "owl:Class"
  "dc11:identifier" "001-BFO"
  "skos:definition" "\"An entity is anything that exists...\"@en"))
#+end_example

From =elot-slurp= two buffer-local hash tables are built:
- =elot-codelist-ht= : id -> label
- =elot-attriblist-ht= : label -> plist-of-attributes

Label-display then uses these tables during font-lock to replace CURIEs
with their human-readable labels via text properties.

** The obsolete =elot-slurp-global=

The current code has a flat in-memory =elot-slurp-global= variable,
populated by =elot-read-slurp-global= from Elisp =.el= files produced
by ad-hoc SPARQL queries. This does not scale (regex-size limit,
no per-project filtering, no priority resolution when sources disagree)
and will be *replaced* by the SQLite-backed store described here.

** Key terms (glossary)

- *CURIE* - compact URI, e.g. =obo:BFO_0000001=. ELOT identifies resources
  by CURIEs throughout.
- *slurp* - the =(ID LABEL PLIST)= data shape above; the common currency
  between parsers, the DB, and lookup code.
- *source* - a file or SPARQL query registered as a provider of slurp
  entries. Identified by its path or URL.
- *active sources* - the subset of registered sources enabled in the
  current buffer/project, in priority order (index 0 = highest).
- *ROBOT* - external Java tool used to execute SPARQL queries against
  OWL files; already integrated in ELOT via =elot-robot-execute-query=.
- *ELOT buffer* - an Org-mode buffer that follows ELOT conventions
  (headlines with =:ID:= properties declaring ontology resources).

** Database schema (authoritative DDL)

#+begin_src sql
PRAGMA foreign_keys = ON;

CREATE TABLE IF NOT EXISTS schema_version (
  version INTEGER PRIMARY KEY
);

CREATE TABLE IF NOT EXISTS sources (
  source        TEXT NOT NULL,
  data_source   TEXT NOT NULL DEFAULT '',  -- '' for org/csv/tsv/ttl; file path or endpoint URL for .rq
  type          TEXT,     -- 'org', 'csv', 'tsv', 'ttl', 'rq'
  last_modified REAL,     -- file mtime at last parse (0.0 for remote queries)
  last_updated  REAL,     -- wall-clock time of last successful update
  PRIMARY KEY (source, data_source)
);

CREATE TABLE IF NOT EXISTS entities (
  id          TEXT,
  label       TEXT,
  source      TEXT NOT NULL,
  data_source TEXT NOT NULL DEFAULT '',
  PRIMARY KEY (id, source, data_source),
  FOREIGN KEY (source, data_source) REFERENCES sources(source, data_source) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS attributes (
  id          TEXT,
  source      TEXT NOT NULL,
  data_source TEXT NOT NULL DEFAULT '',
  prop        TEXT,
  value       TEXT,
  FOREIGN KEY (id, source, data_source) REFERENCES entities(id, source, data_source) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_attrs_id_source ON attributes(id, source, data_source);
CREATE INDEX IF NOT EXISTS idx_entities_id     ON entities(id);
#+end_src

** Prefix tables (authoritative DDL, schema v2)

Introduced by Step 1.1.1. Two tables: =prefixes= is per-source and
cascades with its owning source; =global_prefixes= is a small, user-
extensible set of de facto universal prefixes seeded by =elot-db-init=.

#+begin_src sql
CREATE TABLE IF NOT EXISTS prefixes (
  source        TEXT NOT NULL,
  data_source   TEXT NOT NULL DEFAULT '',
  prefix        TEXT NOT NULL,         -- e.g. 'obo'  (empty string allowed for default ':' prefix)
  expansion     TEXT NOT NULL,         -- e.g. 'http://purl.obolibrary.org/obo/'
  PRIMARY KEY (source, data_source, prefix),
  FOREIGN KEY (source, data_source)
    REFERENCES sources(source, data_source) ON DELETE CASCADE
);

CREATE TABLE IF NOT EXISTS global_prefixes (
  prefix      TEXT PRIMARY KEY,         -- 'rdf', 'rdfs', 'owl', 'xsd', 'skos', 'dcterms', 'foaf', 'obo', ...
  expansion   TEXT NOT NULL,
  trust       INTEGER NOT NULL DEFAULT 1  -- reserved for future weighting; 1 = de facto universal
);

CREATE INDEX IF NOT EXISTS idx_prefixes_expansion ON prefixes(expansion);
#+end_src

*=entities.id= invariant.* =entities.id= holds *the canonical form
the parser saw*: a full URI when the parser was able to resolve the
identifier (declared =@prefix=, configured prefix map, or known
global prefix), otherwise the CURIE literal as written in the
source (e.g. =ex:hello=). The column is never empty. CURIE/URI
forms are *derived* at lookup time by joining against the
applicable prefix tables (source-scoped first, then
=global_prefixes=). The DB never stores =URI= and =CURIE= as sibling
columns; see Decisions Log for why.

An optional companion column =entities.kind= (=TEXT=, values
='uri'= / ='curie'= / ='unknown'=) records which form =id= holds. It
is informational only - useful for debugging and for the lookup
code to skip pointless CURIE-to-URI round-trips - and is not part
of any key.

*Unresolved CURIEs.* Many real-world sources ship CURIEs whose
prefixes cannot be resolved at parse time: SPARQL =.rq= files
without =PREFIX= lines, CSV/Org tables of bare CURIEs, Python code
that uses string CURIEs without an accompanying =Namespace=
binding. The DB *accepts these*: =entities.id= stores the CURIE
literal and =kind = 'curie'=. Label-display still works, because
lookup matches on the literal =id= first. Two consequences worth
naming: (1) if a prefix binding arrives later from another source,
the same resource may end up represented twice (once as CURIE,
once as URI) - v1 accepts this duplication, both rows resolve to
their labels, reconciliation is deferred; (2) different sources
that happen to use the same prefix string for different expansions
will collide on =id= - the source-scoped =prefixes= table and the
priority order of =elot-active-label-sources= keep the right one
winning in a given buffer.

*Two-pass lookup.* The DB-level label lookup
(=elot-db-get-label-any=, defined in Step 1.1.1) does:

1. Try =id= literal match. Covers both stored-as-URI and
   stored-as-CURIE cases without any prefix work.
2. If miss and the queried token *looks like* a CURIE, expand it
   via the active source's =prefixes= rows (then =global_prefixes=)
   and re-lookup as a full URI.
3. If miss and the queried token *looks like* a full URI, contract
   it via the same prefix tables (longest-expansion match) and
   re-lookup as each candidate CURIE.

No further passes; if all three miss, the lookup returns =nil=.

** Update transaction (reference implementation)

#+begin_src emacs-lisp
(defun elot-db-update-source (source data-source type data &optional file-mtime)
  "Replace DB records for (SOURCE, DATA-SOURCE) with DATA.
DATA is a list of (id label plist) triples. DATA-SOURCE is nil for
non-.rq sources; a local file path or endpoint URL for .rq sources."
  (sqlite-transaction elot-db
    (sqlite-execute elot-db
      "DELETE FROM entities WHERE source = ? AND data_source IS ?"
      (list source data-source))
    (dolist (row data)
      (let ((id    (nth 0 row))
            (label (nth 1 row))
            (plist (nth 2 row)))
        (sqlite-execute elot-db
          "INSERT OR REPLACE INTO entities (id, label, source, data_source) VALUES (?, ?, ?, ?)"
          (list id label source data-source))
        (cl-loop for (prop val) on plist by #'cddr do
          (sqlite-execute elot-db
            "INSERT INTO attributes (id, source, data_source, prop, value) VALUES (?, ?, ?, ?, ?)"
            (list id source data-source prop val)))))
    (sqlite-execute elot-db
      "INSERT OR REPLACE INTO sources (source, data_source, type, last_modified, last_updated)
       VALUES (?, ?, ?, ?, ?)"
      (list source data-source type (or file-mtime 0.0) (float-time)))))
#+end_src

** Priority-aware lookup (reference implementation)

The active-sources list is both a *filter* (only these pairs count)
and an *ordering* (earlier = higher priority). Each active entry is a
=(SOURCE DATA-SOURCE)= pair; lookup matches against both columns and
sorts by the entry's position in the list.

#+begin_src emacs-lisp
(defun elot-db--source-key (entry)
  "Return a stable key for active-source ENTRY for comparison."
  (cons (nth 0 entry) (nth 1 entry)))

(defun elot-db--match-clause (entries)
  "Build a SQL WHERE fragment and parameter list matching ENTRIES.
ENTRIES is a list of (SOURCE DATA-SOURCE). Returns (SQL . PARAMS)."
  (let ((clauses nil) (params nil))
    (dolist (e entries)
      (let ((src (nth 0 e)) (ds (nth 1 e)))
        (if ds
            (progn (push "(source = ? AND data_source = ?)" clauses)
                   (setq params (append params (list src ds))))
          (push "(source = ? AND data_source IS NULL)" clauses)
          (setq params (append params (list src))))))
    (cons (mapconcat #'identity (nreverse clauses) " OR ") params)))

(defun elot-db-get-label (id &optional active-sources)
  "Return label for ID, restricted and prioritized by ACTIVE-SOURCES.
ACTIVE-SOURCES defaults to buffer-local `elot-active-label-sources',
whose entries are (SOURCE DATA-SOURCE) pairs."
  (let ((sources (or active-sources elot-active-label-sources)))
    (when sources
      (pcase-let* ((`(,where . ,params) (elot-db--match-clause sources))
                   (query (format
                           "SELECT label, source, data_source FROM entities
                             WHERE id = ? AND (%s)" where))
                   (rows  (sqlite-select elot-db query (cons id params)))
                   (keys  (mapcar #'elot-db--source-key sources)))
        (caar (sort rows
                    (lambda (a b)
                      (< (cl-position (cons (nth 1 a) (nth 2 a)) keys :test #'equal)
                         (cl-position (cons (nth 1 b) (nth 2 b)) keys :test #'equal)))))))))

(defun elot-db-get-attr (id prop &optional active-sources)
  "Return value of PROP for ID, priority-resolved over ACTIVE-SOURCES."
  (let ((sources (or active-sources elot-active-label-sources)))
    (when sources
      (pcase-let* ((`(,where . ,params) (elot-db--match-clause sources))
                   (query (format
                           "SELECT value, source, data_source FROM attributes
                             WHERE id = ? AND prop = ? AND (%s)" where))
                   (rows  (sqlite-select elot-db query
                                         (append (list id prop) params)))
                   (keys  (mapcar #'elot-db--source-key sources)))
        (caar (sort rows
                    (lambda (a b)
                      (< (cl-position (cons (nth 1 a) (nth 2 a)) keys :test #'equal)
                         (cl-position (cons (nth 1 b) (nth 2 b)) keys :test #'equal)))))))))
#+end_src

** Two-tier lookup (local HT first, DB fallback)

These replace the current definitions in =elot-label-display.org=:

#+begin_src emacs-lisp
(defun elot-codelist-id-label (idstring)
  "Return label for IDSTRING: local hash table first, then global DB."
  (or (elot--strip-lang-tag (gethash idstring elot-codelist-ht))
      (and (fboundp 'elot-db-get-label)
           (elot-db-get-label idstring))))

(defun elot-attriblist-label-value (label prop)
  "Return PROP from attribute plist for LABEL: local first, then global DB."
  (or (plist-get (gethash label elot-attriblist-ht) prop 'equal)
      (and (fboundp 'elot-db-get-attr)
           (elot-db-get-attr label prop))))
#+end_src

** Needs-update check (mtime comparison)

#+begin_src emacs-lisp
(defun elot-db-source-needs-update-p (file &optional data-source)
  "Return t if FILE has been modified since last parsed into the DB.
DATA-SOURCE disambiguates .rq sources; nil otherwise."
  (let* ((mtime (float-time (file-attribute-modification-time
                             (file-attributes file))))
         (row   (sqlite-select elot-db
                 "SELECT last_modified FROM sources
                   WHERE source = ? AND data_source IS ?"
                 (list file data-source)))
         (db-mtime (caar row)))
    (or (null db-mtime) (> mtime db-mtime))))
#+end_src

* File & Module Layout

** MELPA alignment (important)

ELOT is being prepared for submission as a MELPA package. The folder
=elot-package/= is the package root: everything shipped to MELPA users
must live there (or be referenced from there by the recipe's =:files=
clause). This constraint drives the layout below.

*Decision:* New Elisp files for the global-slurp subsystem are authored
directly under =elot-package/= from day one, alongside the existing
=elot.el=, =elot-mode.el=, =elot-label-display.el=, =elot-tangle.el=,
=elot-flymake.el=, =elot-lint.el=, and =elot-owl-grammar.el=. We do *not*
start them under a sibling =slurp/= folder and move them later. Rationale:
- Keeps =load-path= stable from the first commit; existing user configs
  and =elot-init.el= already add =elot-package/= to =load-path=.
- Avoids a later file-move churn that would break =git blame=, rewrite
  all =require= paths, and force re-byte-compilation downstream.
- Byte-compilation, =checkdoc=, =package-lint=, and autoload generation
  are already (or will be) wired for =elot-package/=; a second folder
  means a second compile/lint target and a real risk of shipping
  uncompiled or unchecked files.
- Cross-file =require='s (e.g., =elot-label-display.el= calling
  =elot-db-*=) resolve naturally without conditional path handling.

*What stays outside =elot-package/=:*
- *Tests and fixtures* - MELPA packages should not ship their test
  harness. Tests live at a top-level =test/= (see Test Infrastructure).
- *VS Code port* (=vscode-elot/=) - separate tree, unaffected by MELPA.

*What ships inside =elot-package/= besides =.el= files:*
- =schema.sql= - canonical DDL, shared verbatim with the VS Code port.
  Located at runtime via
  =(expand-file-name "schema.sql" (file-name-directory (locate-library "elot-db")))=.
  Included in the MELPA recipe's =:files= clause.

We also *break with the literate-Org pattern* for this subsystem:
=elot-db.el= and =elot-sources.el= are authored directly as Elisp - no
=elot-db.org= literate source file. Each file carries a =;;; Commentary:=
header referencing this plan document so the =slurp/=-as-topical-group
idea is preserved in documentation even without a dedicated folder.

* Portability Boundary

Everything below the line =elot-db-*= API is Emacs-specific (uses
=sqlite.el=). Everything above is pure Elisp working on slurp lists.
The VS Code port reimplements *only* the =elot-db-*= functions in
TypeScript against =sql.js=; parsers are re-authored in TS but produce
the same slurp shape. The =schema.sql= file is shared verbatim.

* Emacs Version Requirements

- *ELOT overall:* Emacs 29+ remains the stated minimum; users may try
  earlier versions at their own risk.
- *Buffer-local slurp* (the existing =elot-slurp= mechanism, populated
  from the current ELOT Org buffer): works on any Emacs that runs ELOT.
  No SQLite dependency.
- *Global slurp* (the new =elot-db= layer, drawing labels from sources
  other than the current Elot Org buffer): *requires Emacs 29+* for
  built-in =sqlite.el=. If SQLite is unavailable, ELOT degrades
  gracefully to local-only behavior (with a one-time warning).

* Status Legend

- =TODO= - not started
- =WIP=  - in progress
- =DONE= - implemented, tested, merged
- =BLOCKED= - waiting on external factor (see notes)

* Guiding Principles

1. *Thin DB abstraction layer.* All SQLite access goes through a small set of functions (=elot-db-*=). Nothing else in the codebase knows the storage backend. This is the seam for the VS Code port.
2. *Local always wins.* Buffer-local =elot-slurp= for the current ELOT file is authoritative for its own identifiers. The DB is a fallback / enrichment source.
3. *Incremental, testable steps.* Each step ends in a working state with passing tests. No "big bang" integration.
4. *Portable SQL.* DDL and query strings written once, reused verbatim by the VS Code port.
5. *Graceful degradation.* If SQLite is unavailable, network is down, or a file is missing, ELOT keeps working on whatever it has.

*Goal:* Add first-class prefix support to the DB so that

   preserved. Return a hashtable =label -> id= (or
   =label -> (id source data-source)= if the annotator needs
   the source). Do not reuse =elot-db-all-active-ids= + N
   round-trips through =elot-db-get-label= - that is O(N) SQL
   and unusable past a few thousand entities.

2. /Replace the insert step./ The current path inserts the
   ="puri"= slot synthesised by slurp; that slot does not exist
   in the DB. Compute the token to insert at selection time:
   #+begin_src elisp
   (let* ((id     (gethash selected-label label->id))
          (curies (and (elot-db--looks-like-uri-p id)
                       (elot-db-contract-uri id elot-active-label-sources)))
          (token  (or (car curies) id)))
     (insert token))
   #+end_src
   This handles both the full-IRI case (contract via active
   plus global prefixes) and the already-CURIE case (ids
   stored as CURIEs - insert verbatim).

3. /Generalise the annotation function./ Factor the existing
   padding/format block in =elot-label-lookup-annotations= out
   into a private helper that takes =(label prefix rdf-type
   definition)= and returns the formatted annotation string.
   The slurp-backed annotator (the existing body) and the new
   DB-backed annotator both call this helper. The DB annotator
   fetches attributes via =elot-db-get-all-attrs id
   elot-active-label-sources= and reads
   =rdf:type= / =iof-av:naturalLanguageDefinition= /
   =skos:definition= / =dcterms:description= / =rdfs:comment=
   in the same priority order used today.

4. /Dispatch on context./ Turn =elot-label-lookup= into a thin
   wrapper:
   #+begin_src elisp
   (defun elot-label-lookup ()
     (interactive)
     (cond
      ((and (bound-and-true-p elot-attriblist-ht)
            (hash-table-p elot-attriblist-ht)
            (> (hash-table-count elot-attriblist-ht) 0))
       (elot-label-lookup--from-attriblist))
      ((and (bound-and-true-p elot-active-label-sources)
            (fboundp 'elot-db-all-active-labels))
       (elot-label-lookup--from-db))
      (t
       (user-error
        "No ELOT labels available: no slurp data and no active DB sources"))))
   #+end_src
   Order matters: prefer the slurp cache inside ELOT buffers
   (it carries richer attributes like
   =iof-av:naturalLanguageDefinition= that are not in the DB
   unless ingested); fall back to DB elsewhere.

*Decisions (from the Q&A that produced this step):*
- /Label collisions across sources./ Keep the highest-priority
  result silently. This matches the Step 1.11 decision (1) for
  =elot-db-get-label-any= and the documented /Priority matters/
  rule in the README. Concretely:
  =elot-db-all-active-labels= walks =elot-active-label-sources=
  in order and skips labels that are already in the result
  hash.
- /No active sources./ =user-error= with a message pointing at
  the source-list UI - same style as
  =elot-global-label-display-mode='s no-sources message
  (Step 1.11 decision 2), but this is a command not a mode, so
  =user-error= is appropriate here rather than =message=.
- /Autoload./ Add =;;;###autoload= to =elot-label-lookup= so a
  user who has installed ELOT but has never opened an ELOT Org
  file can still run =M-x elot-label-lookup= from, e.g., a
  plain Python buffer. Transitively this means
  =elot-label-display.el= and =elot-db.el= must be loadable
  standalone - verify no hard top-level references to Org or
  ELOT-Org machinery at load time (runtime guards are fine).
- /Keybinding./ The existing =C-c C-x r= dispatcher in
  =elot-defs.org= is bound in =elot-mode-map=; also bind
  =elot-label-lookup= in =elot-global-label-display-mode-map=
  (added in Step 1.7) so the keybinding follows the feature
  into non-ELOT buffers.
- /Performance./ =elot-db-all-active-labels= will run once per
  invocation; cache per =(sources . db-mtime)= only if it
  becomes noticeable. Not needed for v1.

*Goal:* Make =elot-label-lookup= correct and usable when a
single label is borne by many identifiers in an external
source.  The motivating case is industrial-asset =.ttl= files
where e.g. 20 distinct instances all carry =rdfs:label "thimble

hash that silently drops 19 of the 20 rows; the user can only
ever insert one id per label and has no way to see the others.

*Scope decision (per user):* We treat the multi-id problem as
an *external-sources-only* concern.  Duplicate labels /within/
an ELOT Org file are a data-quality issue that =elot-lint= and
related checks surface elsewhere; we do not plumb

*Design:*

- Data-layer change.  =elot-db-all-active-labels= is widened to
  return =label -> list-of-ids= (ordered by source priority,
  then by id).  All existing single-id callers take =car= of
  the list.  The SQL is unchanged; only the Elisp return shape
  widens.  A matching helper =elot-db-ids-for-label LABEL= is
  added for convenience (returns the list directly).
- Collection builder (external path).
  =elot-label-lookup--collect-db= now emits one /stage-1 entry/
  per unique =(label, source)= pair, carrying =:ids= (the list)
  and =:count= (its length).  When =count = 1=, the entry
  retains the full attribute plist for the sole id so its
  annotation is as rich as today.  When =count > 1=, the
  stage-1 annotation shows =N matches= alongside the source
  provenance; attribute fetch is deferred to stage 2.
- Stage 1 (label picker).  Single =completing-read= over the
  unique =(label, source)= entries.  Annotator column shows
  either the usual type/definition (for singletons) or
  =N matches   [src]= (for groups).  Selection returns the
  entry.

  we do not re-enter stage 1.
- Single-id fast path.  When =count = 1=, stage 2 is skipped
  entirely and the id is inserted directly.  This preserves
  today's one-keystroke behaviour for the common case.
- Prefix-arg escape hatch.  =C-u C-u= on =elot-label-lookup=
  forces the /flat/ shape: every id is a stage-1 entry with a
  disambiguating CURIE/IRI-tail suffix, no stage 2.  This

  - Alternative considered: reuse =C-u C-u= for scope=external

*** DONE Step 1.16 - Language-tag-aware label and annotation selection
:PROPERTIES:
:STATUS: TODO
:END:

*Goal:* When an ontology carries labels (and annotations like
=skos:definition=, =rdfs:comment=) in multiple languages, ELOT
must present a /consistent/ language per user preference, not a
chaotic mix.  The motivating case: a source with both English
and Korean labels currently yields arbitrary per-row choices in
label-display overlays, eldoc/hover summaries, and
completion-read candidates, because =elot-db-get-label= and
=elot-db-get-all-attrs= have no language-awareness.

*Scope:* All user-facing surfaces that render labels or
language-tagged annotation values derived from the DB:

- =elot-db-get-label= (primary) - label-display overlays,
  stage-1 completion candidate display.
- =elot-db-get-all-attrs= - eldoc / hover summaries, stage-2
  annotator.
- =elot-db-all-active-labels= - completion key set in
  =elot-label-lookup=.
- =elot-global--eldoc-function= / =elot-global--help-echo= -
  must pick definition in the preferred language.

The slurp path (=elot-attriblist-ht=) is out of scope: ELOT Org
files are authored single-language per-file by convention, and
language tagging there is driven by =elot-default-language=
elsewhere.  If future demand emerges we revisit.

*Default selection policy (when user has specified no
preferences):*

For a given =(id, predicate)= in the DB that has multiple
values differing only in language tag, the winning value is
chosen by:

1. Untagged value (no language tag at all) - preferred.
2. Otherwise, =@en= - preferred as least-surprise default.
3. Otherwise, the value whose language tag sorts first
   alphabetically.

Rationale: untagged beats tagged because authors who bother to
omit the tag usually mean "the canonical form".  =@en= as
default second choice matches the dominant convention in public
ontologies (BFO, IOF, SKOS cores) and the expectation of most
English-speaking users; =ko= or =de= users will set a
preference anyway.

*User-specified policy:*

A new defcustom =elot-preferred-languages= holds an ordered
list of language tags, e.g. =("ko" "en")=.  Empty list
(default) triggers the default policy above.  Interpretation:

- Walk the list in order; first language tag that has a value
  for the =(id, predicate)= wins.
- *Untagged values are included implicitly at the end of the
  preference list by default.*  This means a user who sets
  =("ko")= gets Korean when available, then falls back to
  untagged, then falls back to any remaining language
  (alphabetical).  Users who want to /exclude/ untagged from
  the fallback can use the sentinel symbol =:untagged= anywhere
  in the list to pin its position explicitly (e.g.
  =("ko" :untagged "en")= means "prefer Korean, then untagged,
  then English, then alphabetical").
- The same preference list applies to *all* language-tagged
  string attributes, not just =rdfs:label=.  So a user with
  =("ko")= sees Korean label /and/ Korean definition in eldoc
  when both are present.

*Configuration surfaces:*

- =defcustom elot-preferred-languages= in =elot-db.el= (or
  =elot-sources.el= - TBD during implementation; =elot-db.el=
  is more natural since the selection happens at DB-query
  time).  Type: =(repeat (choice string (const :untagged)))=.
  Default: =nil= (use default policy).
- =.dir-locals.el= support: the variable is marked
  =safe-local-variable= with a predicate accepting lists of
  strings and the =:untagged= keyword.  Projects can pin
  language preferences per project root.
- Not stored in the SQLite DB.  Rationale: this is a user
  presentation preference, not ontology metadata; it belongs
  with other per-user/per-project Emacs configuration.
- Buffer-local override (optional, low priority): a command
  =elot-set-buffer-languages= for interactive experimentation.
  Implement only if users ask.

*Data-layer changes (schema v3 - required):*

/Correction from an earlier draft of this step:/ the current
schema does *not* carry language tags.  =attributes= has columns
=(id, source, data_source, prop, value)= only; =entities= has
=(id, label, source, data_source, kind)=.  Neither table
records the =@tag= from an RDF literal.  A schema bump is
required.

Concretely:

- =ALTER TABLE attributes ADD COLUMN lang TEXT NOT NULL DEFAULT ''=.
  Empty string is the untagged sentinel, matching the =data_source=
  convention elsewhere in the schema.
- =entities.label= stays (denormalised cache of the default label,
  preserved for backward compatibility and for sources that
  contribute exactly one label per id).  Multi-language labels
  are written as additional rows in =attributes= with
  =prop = 'rdfs:label'= and populated =lang=.  =elot-db-get-label=
  queries =attributes= first (language-preference-aware), then
  falls back to =entities.label=.  This avoids a second label
  column on =entities= and keeps the schema change minimal.
- Bump =elot-db-schema-version= from 2 to 3.  Add an
  =elot-db--migrate-v2-to-v3= function using the same
  =elot-db--attributes-has-lang-p= pattern as the existing v1->v2
  path.  Backfill: existing rows acquire =lang = ''= via the
  column default; no data is lost.
- Index: =CREATE INDEX idx_attrs_prop_lang ON attributes(prop, lang)=
  helps the label-by-language read path when a source has many
  =rdfs:label= rows.

*Selection logic on read:*

Fetch-all + pick-in-Elisp via a pure helper
=elot-db--select-by-language ROWS &optional PREFS=, taking a list
of =(value . lang)= pairs.  The SQL side stays boring;
realistic ontologies have O(few) language variants per triple,
so the row count per query is small.  SQL-side ordering (via a
=CASE= built from the preference list) is the escape hatch if
profiling ever surfaces a problem.

All four affected DB read functions route through the helper.

*New helpers:*

- =elot-db--select-by-language ROWS &optional PREFS= - core
  language picker.  Pure function, no DB access, easy to unit
  test in isolation.
- =elot-db--effective-language-prefs= - resolves buffer-local
  and defcustom into an ordered list, appending =:untagged= if
  the user's list doesn't mention it and implicit-tail policy
  is in force.

*Affected surfaces (read path):*

- =elot-db-get-label= - widen internal query to consult
  =attributes WHERE prop='rdfs:label'= first (returning
  =(value, lang)= rows), apply =elot-db--select-by-language=,
  fall back to =entities.label= on miss.
- =elot-db-get-all-attrs= - for each distinct =prop=, collapse
  multi-language rows via the helper before returning the
  plist.  Rows without a language tag (URIs, numeric literals)
  bypass the picker by construction (their =lang= is =''=, which
  slots naturally into the preference chain).
- =elot-db-all-active-labels= - applies the picker per-id when
  building the =label -> ids= map.  Critical for stage-1
  completion stability: two entries in different languages must
  not both reach the completion list.
- Label-display overlay generation / eldoc / hover - unchanged
  structurally; all go through =elot-db-get-label= or
  =elot-db-get-all-attrs=, which now return the language-resolved
  form.

*Affected surfaces (write path / ingestors):*

An audit of the ingestion code (Step 1.16 preflight) shows:

- *TTL via ROBOT* (=elot-source-parse-ttl=, =elot-source.el=):
  the built-in SPARQL query
  =elot-source-ttl-label-query= projects =?id ?label= only.  When
  ROBOT exports to CSV, language tags on =?label= are /dropped/ -
  the CSV carries only the lexical form.  To preserve tags, the
  default query must change to project =?label= and =LANG(?label)=
  as separate columns (e.g. =?id ?label ?lang=), and the CSV
  parser must recognise the =lang= column as the tag for that
  row.  Users who have customised =elot-source-ttl-label-query=
  keep working (language-less, as today) until they opt in.
- *.rq via ROBOT* (=elot-source-parse-rq=): users write their
  own queries; if they want language-tagged labels they add
  =LANG(?label)= themselves.  Documented, not enforced.  The
  CSV parser's lang-column recognition covers both TTL and RQ.
- *CSV / TSV / JSON* (=elot-source-parse-csv=, =-tsv=, =-json=):
  plain text sources have no inherent language concept.  Two
  conventions will be supported:
  1. A dedicated header column named =lang= (case-insensitive)
     gives the tag for the row's label.  If the column is
     absent, all rows are untagged.
  2. Column-suffix form: a header =label@ko= contributes a
     =rdfs:label= attribute row with =lang='ko'=.  Multiple
     such columns per row are allowed and produce one attribute
     row each.
  Backward-compatible: sources without =lang= or =label@*=
  columns ingest exactly as before.
- *Org / ELOT* (=elot-source-parse-org=): out of scope for 1.16
  per the scope decision above.  =elot-build-slurp= is
  untouched.

*Parser contract change (minor, backward-compatible):*

The existing shape is =(ID LABEL PLIST)= (or the extended
=(:entries ... :prefixes ...)= form from Step 1.7.3).  To carry
multi-language labels without a second refactor later, the PLIST
is widened: in addition to existing ="prop" "value"= pairs,
parsers may emit ="prop" (list "value" "lang")=  entries

/Risk/

Medium.  Unlike earlier steps this one touches the schema and
every ingestor, not just read paths.  Specifically:

- Schema migration: the v2->v3 path is mechanically identical to
  the existing v1->v2 path (ADD COLUMN with DEFAULT), but it is
  the first migration to land after users have real data.
  Mitigation: the existing =elot-db-migrate= harness is already
  exercised; the migration is additive (no data rewritten); on
  failure, =sqlite-open= can be re-pointed at a backup.  Worth
  adding a one-liner recommending users back up
  =elot-db-file= before upgrading.
- ROBOT query change: users who have customised
  =elot-source-ttl-label-query= will not get language tags until
  they update their query.  Documented in the docstring and
  README; no silent breakage (old queries still parse, just
  ingest language-less labels).
- Performance: if a source has tens of thousands of
  language-tagged rows per predicate, the "fetch all, pick in
  Elisp" approach could show up in profiles.  Unlikely in
  practice (realistic ontologies are O(few) tags per triple);
  fallback to SQL-side ordering is available if needed.
- Discoverability: users confronted with a Korean label when
  they expected English need a clear pointer to
  =elot-preferred-languages=.  Addressed by the README section
  and a brief note in the eldoc summary when the mode is newly
  activated (TBD, non-blocking).

/Implementation order rationale:/

- /6 -> 7/: CSV first (simpler, no ROBOT dependency), then TTL.
  TTL relies on the CSV-side =lang= column recognition shipped

- /8, 9/: polish and docs, no functional risk.

/Rollback posture:/ sub-steps 1.16.1 through 1.16.4 are all
side-effect-free with respect to user behaviour.  If 1.16.5

** Stage 2: VS Code / CLI Port

*Framing (revised 2026-04-24, after Stage 1 complete).* The original
Stage 2 sketch was drafted when the Elisp surface was Step 1.1 only.
Stage 1 has since grown to ~20 public functions, schema v3, a
scope-aware two-stage lookup, attribute-driven hover, and
language-tag preferences. The rewrite below reflects that reality
and incorporates explicit user decisions made after the Stage 1
review:

- /Separate databases./  Emacs and VS Code each own their own SQLite
  file, in the location that is natural for their platform.  Users
  overwhelmingly live in one editor or the other; cross-editor
  sharing is explicitly not a goal.  No concurrent-write handling
  needed; single-user, single-writer assumed throughout.
- /VS Code is a reader for content, a switch for activation./  The
  extension does not implement =registerSource= / =refreshSource= /
  =removeSource= in its UI.  All DB management (register, refresh,
  remove, prefix add, migrate) is done by the =elot-cli= Node
  binary.  The VS Code UI limits itself to: hover, decorations,
  label lookup (insert id), and activating / deactivating sources
  that already exist in the DB.
- /sql.js is acceptable./  In-memory DB + full-file rewrite on save
  is fine at the corpus sizes we see (tens of MB on disk ->
  hundreds of MB RAM worst case).  Avoiding native prebuilds
  across VS Code versions is worth more than the RAM.
- /Schema v3 only./  No pre-v3 compatibility in the TS port.  The
  DDL is extracted from Elisp into a shared =schema.sql= snapshot;
  both the Elisp loader and the TS loader read it verbatim.
- /Multi-id UX in VS Code./  Flat =QuickPick= with CURIE-suffix
  disambiguation (Step 1.15's =C-u C-u= flat-presentation path,
  not the two-stage default).  VS Code's =QuickPick= does not
  chain naturally; the flat list is the better fit.
- /ELOT Org buffers in VS Code./  The extension already has an Org
  parser (=parseOrgWasm.ts= + =buildSlurp.ts=) that builds a local
  id -> label map for open =.org= files.  The scope-aware union
  (Step 1.13) ports naturally: in =.org= buffers the VS Code
  providers union the buffer-slurp with the DB (local wins on id
  collision); in non-Org buffers only the DB is consulted.
- /elot-cli writes the VS Code database./  The CLI's default DB
  location is the extension's =globalStorageUri= path, i.e. the
  directory VS Code already allocates per-extension for persistent
  data.  An =--emacs= flag (or an =ELOT_DB_PATH= env var) lets a
  power user override to any file; the Emacs side has its own
  =elot-db-file= defcustom and is unaffected.
- /VS Code opens the DB read-only; only the CLI writes./  This is
  the key simplification that falls out of the bullets above.  The
  extension's sql.js handle is loaded from disk and never written
  back: no =SaveChanges=, no =exec('INSERT ...')=, no schema
  creation path in TS.  All mutation (register / refresh / remove /
  prefix / migrate) goes through =elot-cli=, which is the sole
  writer of the VS Code DB file.  Consequences:
  - No sql.js full-file rewrite on save inside the extension;
    the RAM-cost concern from the earlier bullet applies only to
    the read-side snapshot.
  - No concurrent-writer problem even in principle: VS Code and
    the CLI cannot both be writing, because VS Code never writes.
    If the CLI refreshes while the extension is open, the
    extension picks up the new file on its next reload
    (=globalStorageUri= watcher or an explicit "reload DB"
    command - see Step 2.5).
  - The TS port only needs =SELECT= paths.  =schema.sql=,
    migrations, and the loader's INSERT logic stay Elisp-/CLI-only
    and never need TS equivalents.
  - Testing surface shrinks accordingly: the extension's DB tests
    are fixture-driven (open a pre-built =.sqlite=, assert
    lookups), with no write-path coverage required.

3. Platform-specific VS Code global storage directory, under a

   global default.

HT regexes), whose matcher is built from the buffer's own
=elot-slurp=. URIs and CURIEs both work there today; adding heavy

a point query and carries no regex cost. Confirmed empirically on
=examples/bfo-fortest.org= with 4 active sources including a
1361-IRI TTL file.

rejected:

- Fragile: renaming or removing a prefix silently misresolves all
  stored ids that used it.
- Inconsistent with what ROBOT emits -- would require a
  post-processing pass on every TTL ingest.
- Breaks cross-source identity: two TTL files that both mention

* Decisions Log

Append-only record of architectural decisions, newest first. Each entry:
date, decision, rationale.

- *=entities.id= holds URI-if-known-else-CURIE-literal:* the
  invariant for =entities.id= is "the canonical form the parser
  saw" - a full URI when an expansion for the prefix was available
  at parse time (declared =@prefix=, configured prefix map, known
  global prefix), and the CURIE literal otherwise. The column is
  never empty. An optional =entities.kind= column ('uri' / 'curie'
  / 'unknown') records which form was stored. Rationale: refusing
  to ingest sources whose CURIEs cannot be resolved would strip
  label-display of exactly the users it most helps - SPARQL
  =.rq= files without =PREFIX= lines, CSV/Org tables of bare
  CURIEs, Python source using string CURIEs without an
  accompanying =Namespace= binding. The literal-id match is the
  first pass of =elot-db-get-label-any= and works without any
  prefix machinery, so the valuable case (label-display over an
  unresolvable CURIE) is supported by construction. *Known
  limitation (v1):* if a prefix binding arrives later from a
  different source, the same resource may end up stored twice
  (once as CURIE, once as URI). v1 accepts the duplication; both
  rows resolve to their labels via the two-pass lookup.
  Reconciliation strategies (rewrite-on-binding-insertion;
  query-time merge) are deferred until users hit the case.
- *Prefix support is two tables, not =URI=/=CURIE= sibling columns:*
  added in schema v2 (Step 1.1.1) as =prefixes= (per-source,
  cascading) plus =global_prefixes= (small, user-extensible,
  seeded at init). =entities.id= remains the canonical form (URI
  when known, CURIE literal otherwise; see preceding entry); CURIE
  forms for a stored URI are derived at lookup time. The rejected
  alternative was to add =URI= and =CURIE= columns to =entities=
  alongside =id=. Reasons for rejection: (1) redundant storage
  (every row carries the full prefix expansion, costly for OBO /
  SNOMED-scale ontologies); (2) update anomalies if a prefix
  expansion is later changed (rewrite every row vs. update one
  prefix row); (3) ambiguity when a URI has multiple valid CURIE
  forms (e.g. =obo:RO_0002160= vs =RO:0002160= via the "banana"
  convention) - two columns force one answer, a prefix table lets
  both coexist and the renderer choose; (4) =id= becomes
  ambiguous about which form it holds; (5) prefixes are a property
  of *sources*, not entities - a 1-to-many =sources -> prefixes=
  relationship that entity-level columns cannot scope. The
  source-scoped vs. global split mirrors how prefixes are used in
  practice: project-level bindings are local; =rdf=, =rdfs=, =owl=,
  =xsd=, =skos=, =dcterms=, =foaf=, =obo=, =sh=, =prov= are de
  facto universal and worth ambient availability. Conflict
  resolution: source-scoped =prefixes= rows win over
  =global_prefixes=; among source-scoped rows, the priority order
  of =elot-active-label-sources= breaks ties. Round-trip ambiguity
  for URI-to-CURIE contraction is resolved by longest-expansion
  first, then active-source priority. The =data_source= column on
  =prefixes= follows the =NOT NULL DEFAULT ''= convention for the
  same =MATCH SIMPLE= reason as the other tables.
- *=data_source= is =NOT NULL DEFAULT ''= on all three tables:* in
  =sources=, =entities=, and =attributes=, the =data_source= column is
  declared =TEXT NOT NULL DEFAULT ''=. The empty string is the sentinel
  for "no sub-dataset" (i.e. for org/csv/tsv/ttl sources and for =.rq=
  sources without a separate data target, should any arise). Rationale:
  SQLite's composite-FK semantics default to =MATCH SIMPLE=, under which
  a child row whose FK contains *any* =NULL= is considered to reference
  no parent - so =ON DELETE CASCADE= silently skips it. =MATCH FULL= is
  parsed but not implemented by SQLite, so the only reliable fix is to
  forbid =NULL= in the FK columns. Using =''= rather than, say, =-= or
  =none= keeps the sentinel visually unobtrusive in dumps and joins
  naturally with omitted columns on insert. Callers may omit
  =data_source= entirely when inserting non-=.rq= rows; the default
  fills it in. This supersedes the earlier "nullable =data_source="
  decision below; for =.rq= rows the column simply carries the query's
  data source as before.
- *SPARQL =.rq= sources are two-part (query + data source):* a =.rq=
  source is identified by the *pair* =(query-file, data-source)=,
  where =data-source= is either a local RDF file (passed to ROBOT as
  =--input=) or a SPARQL endpoint URL (used via =SERVICE= federation
  with a dummy input). The same query may appear more than once with
  different data sources, and vice versa. Rationale: both local-file
  querying (often the most useful ELOT case) and remote federation
  are first-class; a single-file model cannot express this.
- *Schema carries a nullable =data_source= column:* the =sources=
  table's primary key is =(source, data_source)=; =entities= and
  =attributes= carry the same pair as a composite foreign key. For
  non-=.rq= rows, =data_source= is =NULL=. Rationale: minimal-change
  extension that uniquely identifies every source; avoids a separate
  =rq_sources= table and its join cost.
- *Project source list is a two-column ordered list (no priority
  integer):* =elot-active-label-sources= holds entries of the form
  =(SOURCE DATA-SOURCE)= (with =DATA-SOURCE= =nil= except for =.rq=).
  Priority is the list order - first entry wins. Rationale: row
  order in =.dir-locals.el= is naturally linear and self-evident;
  an explicit integer is redundant and invites stale or conflicting
  values.
- *=.rq= execution always routes through ROBOT:* both local-file and
  remote-endpoint =.rq= sources are executed via
  =elot-robot-execute-query= with =format = "csv"=. Remote endpoints
  are reached by =SERVICE <url>= federation (either embedded in the
  user's query or via a generated wrapper), with a dummy/empty input
  ontology for ROBOT. Rationale: one executor codepath, matching the
  existing ROBOT integration; no additional HTTP stack to maintain.
- *=.rq= result caching under =.elot-cache/=:* the CSV output of each
  =.rq= execution is cached in =.elot-cache/<query-basename>.<hash>.csv=
  at the project root, where =<hash>= is a short digest of
  =data-source=. This disambiguates the common case of a single query
  used against multiple data sources. Refresh is triggered only by
  mtime staleness (query or local data-source newer than cache) or
  explicit user request; empty results and executor errors preserve
  the existing cache. No auto-refresh on activation. Rationale: keeps
  slow ROBOT invocations out of the interactive path; makes caches
  trivially =.gitignore=-able as a single directory.
- *Language-tag handling at write time (v1):* when a source's primary
  label carries a language tag (e.g. ='entity'@en=), the stripped
  string =entity= is stored in =entities.label=, and a single row is
  written to =attributes= with =prop = "elot:labelLang"= and
  =value = "en"=. Alternative-language labels are *not* stored in v1.
  Rationale: keeps the write path simple and the =entities.label=
  column free of tags (matches the earlier "store without language
  tag" decision); multilingual label support can be layered on later
  without a schema change.
- *No CI / GitHub Actions for this plan:* testing is local-only on
  Linux with Emacs 30. A multi-version / multi-OS matrix is
  explicitly out of scope. Rationale: keeps the scope of Stage 1
  bounded and avoids committing to infrastructure that is not
  needed for the feature to land.
- *Schema version seeded at init:* =elot-db-init= writes =version = 1=
  into =schema_version= on a fresh database, and calls =elot-db-migrate=
  (stub) against any pre-existing database; mismatch is a hard error,
  never a silent rewrite.
- *=elot-db-init= accepts an optional path argument:* makes temp-DB
  use in tests straightforward without =let=-binding the defcustom.
- *Editing literate-Org sources vs. tangled Elisp:* modifications to
  existing ELOT modules happen in the =.org= sources and are picked
  up by the normal tangle flow. Only the *new* files introduced by
  this plan (=elot-db.el=, =elot-sources.el=) break the literate-Org
  convention and are authored directly as Elisp. Rationale: minimize
  disruption to the existing codebase while keeping the new, SQL-
  heavy module simple.
- *Folder layout:* new Elisp files are authored directly under
  =elot-package/= (the MELPA package root), *not* under a sibling
  =slurp/= folder. Rationale: ELOT is being prepared for MELPA
  submission; keeping package files co-located from day one avoids a
  later file-move, preserves =git blame=, keeps =load-path= stable
  (users already have =elot-package/= on it), and aligns with the
  existing byte-compile / lint toolchain. Tests and fixtures live
  outside =elot-package/= (under top-level =test/=) so they are not
  shipped to package users. The topical-grouping intent previously
  captured by the =slurp/= name is preserved through the =elot-db-=
  / =elot-sources-= filename prefixes and each file's =;;; Commentary:=
  header.
- *No literate Org for elot-db:* =elot-db.el= and =elot-sources.el=
  are written directly as Elisp, breaking the literate-Org convention
  used elsewhere in ELOT. Rationale: user preference; keeps the
  SQL-heavy module simpler and the test loop tighter.
- *Retire =elot-slurp-global=:* removed outright (with one release
  cycle of obsolescence shims). ELOT remains fully usable with only
  buffer-local slurp. Global slurp via =elot-db= requires Emacs 29+
  for SQLite; ELOT's overall minimum stays at Emacs 29 with earlier
  versions unsupported but not forbidden.
- *DB location:* =elot-db-file= defaults to
  =(locate-user-emacs-file "elot-cache.sqlite")=. Cross-editor DB
  sharing with VS Code is explicitly out of scope for now.
- *Schema migration:* automatic migration is not performed. Users
  must invoke =elot-db-migrate= explicitly when a schema version
  mismatch is detected. Rationale: predictability and safety; avoids
  surprise rewrites of the user's cache on version bumps.
- *CSV/TSV column mapping:* fixed as =col1=id, col2=label, cols3+=
  attributes keyed by header name=. No configurable mapping at this
  stage.
- *SPARQL result shape contract:* any user-supplied SPARQL query used
  as a label source *must* return at least two columns - the first
  with URI/CURIE content (string or URI datatype), the second with a
  literal label (string datatype). Additional columns are treated as
  optional attributes (column name -> plist key).
- *Primary label not duplicated in attributes:* =rdfs:label= (or the
  chosen primary label property) populates =entities.label= only; it
  is not also written to =attributes=. Alternative / language-tagged
  labels may still go into =attributes=.
- *Language tags:* labels are stored in =entities.label= *without*
  language tag, to optimize retrieval (no strip-at-lookup cost). When
  the preferred label carries a language tag, the tag is recorded as
  an attribute row rather than embedded in the label string.
- *Empty active-sources list:* if =elot-active-label-sources= is empty
  in the current buffer, no DB lookup is performed and no labels are
  produced by the global layer. This is not an error.
- *v1 matching is literal tokens only (line in the sand):* the
  font-lock matcher built in Step 1.7 looks for *literal* =id=
  strings from the DB, anchored by word/symbol boundaries, nothing
  more. Explicitly *out of scope for v1*: regex-captured identifiers
  (e.g. =#1234= with the digits looked up after a capture), per-source
  display templates, partial-string substitution inside larger
  tokens, multi-label-per-id with language selection, language- or
  major-mode-specific tokenisers, and identifier alias resolution
  beyond the two-pass URI/CURIE lookup already defined in Step 1.1.1.
  Rationale: once UC3 is blessed, requests of this shape are
  predictable and each is individually reasonable; committing to
  literal-only matching now keeps v1 small, keeps the DB-driven
  regex construction tractable, and provides a documented baseline
  against which future proposals must justify themselves.
- *Subsystem is general-purpose; ELOT is the first consumer (UC3
  blessed):* the global-slurp subsystem intentionally supports three
  peer use cases - UC1 (ELOT author), UC2 (cross-editor ontology
  browsing), and UC3 (generic =id -> label= mapping in projects with
  no ontology involvement). The non-goal previously phrased as "not
  a general-purpose identifier registry" was too narrow and has been
  revised; the real limits are single-user, single-machine, non-
  authoritative, non-triplestore, which remain in force. No rename
  of the =elot-*= symbol space is undertaken; the standalone story is
  a documentation concern (Step 1.8) only.
- *Parser dispatcher is user-extensible:* =elot-source-supported-extensions=
  is a =defcustom=, not an internal constant, so users can register
  project-specific parsers (JSON variants, YAML, XML, domain-specific
  formats) without patching ELOT. Low-cost lever for UC3.
- *JSON parser is in v1:* a simple JSON-dict parser (flat or
  =label=-nested) ships in Step 1.3. Rationale: flat =id -> label=
  JSON dicts are the most common ad-hoc mapping shape in non-RDF
  projects (config, i18n, dashboard legends); CSV alone forces users
  to convert, and they won't. Adding JSON at the time CSV/TSV are
  added costs very little and dramatically broadens UC3 coverage.
