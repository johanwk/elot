* Purpose and how to use this document

This document is the working plan for getting ELOT (=elot-package/=)
into shape for submission to MELPA.  It is structured for incremental
execution with LLM assistance through Emacs ~gptel~:

- Each *Milestone* is a self-contained unit of work.  Open the
  milestone heading, narrow to it (=C-x n s=), and send the whole
  subtree to ~gptel~ as the prompt.
- Each milestone lists *Files affected* and *Steps* as concrete
  verbs ("edit X to do Y") — not pseudocode, not snippets — so
  gptel has unambiguous targets.
- Each milestone ends with an *Acceptance check*: a short list of
  shell or Emacs commands whose output proves the milestone is done.
  After every milestone the package must remain in a *stable status*:
  byte-compiles cleanly, the existing ERT suite passes, =elot-mode=
  still loads and enables in an example file.
- Milestones are ordered to minimise rework.  The SPARQL rewrite
  comes first (it changes the shape of code that later milestones
  touch), then legacy cleanup, then metadata polish, then CI and
  the recipe.

Conventions used below:
- =MAIN FILE= refers to the file declared as the package's
  top-level descriptor.  *The MAIN FILE is
  =elot-package/elot.el=* and the MELPA package name is =elot=.
  This matches MELPA's convention that, for a recipe named
  =elot=, the file =elot.el= is treated as the main library.
  In addition to carrying the canonical headers, =elot.el= is the
  single load entry point: it ~require~s the other =elot-*.el=
  modules so that a user only needs =(require 'elot)= (or
  ~load-library~ "elot") to make =elot-mode=, =elot-tangle= and the
  rest available.
- "Stable status" is the green checkpoint after each milestone:
  see [[*Stable-status checklist (run after every milestone)][Stable-status checklist]].

#+begin_src sh

#+end_src

This runs byte-compile, the ERT suite, and the load smoke test in
sequence, writes per-step logs under =tmp/stable-check/=, and prints
a one-line =PASS= / =FAIL= summary.  Use the individual targets
below when iterating on a single check.

1. *Byte-compile is clean* — =make byte-compile= (or, equivalently,
   from the repository root):
   #+begin_src sh
   cd elot-package
   emacs --batch -L . \
     --eval "(setq byte-compile-error-on-warn nil)" \
     -f batch-byte-compile *.el
   #+end_src
   No errors.  Warnings should not increase relative to the previous
   stable point (record the baseline count once at the start; see
   Step B.1 / =make baseline-warnings=).

2. *Tests pass* — =make test= (which delegates to =make -C test=):
   #+begin_src sh
   make -C test
   #+end_src
   All ERT tests green (ROBOT-gated tests may skip; that is
   acceptable and expected on machines without ROBOT).

3. *Loads in a fresh Emacs* — =make smoke=:
   #+begin_src sh
   emacs --batch -L elot-package \
     --eval "(require 'elot-mode)" \
     --eval "(message \"OK: elot-mode loaded\")"
   #+end_src
   Note: until [[#milestone-4-package-requires][Milestone 4]] wires =elot.el= up as the canonical
   load entry point, the smoke test requires =elot-mode= directly.
   From Milestone 4 onwards (specifically Step 4.6), the smoke test
   switches to =(require 'elot)= and additionally asserts that
   =elot-mode= is bound (autoloaded) as a side effect.  The
   =make smoke= target in the top-level =Makefile= is updated at
   that point.

4. *Working tree clean* — commit the milestone before moving on:
   =git status= shows no uncommitted changes after the commit.

* DONE Milestone 1 — Rewrite SPARQL support
:PROPERTIES:
:CUSTOM_ID: milestone-1-sparql
:END:

** Background

ELOT customises Org's ~org-babel-execute:sparql~ (defined in
=elot.el= ≈ line 513, installed by =advice-add= ≈ line 545) to:

- inject prefix declarations from =org-link-abbrev-alist-local=
  into every query, and
- support querying local RDF/OWL files via ROBOT — which is the
  most common ELOT workflow: querying the ontology you are
  currently editing.

The current implementation has three weaknesses that should be
fixed *before* the rest of the submission prep, because subsequent
milestones (legacy LOB removal, mode lifecycle wiring, header
polish) all touch this code:

1. The ELOT advice builds the prefix block from
   =org-link-abbrev-alist-local= via =elot-prefix-block-from-alist=
   (=elot.el= ≈ line 530). A query body without any PREFIX boilerplate
   must execute correctly. (This does seem to work OK already.)

2. *Robustness gaps.*  The advice does not gracefully handle:
   - missing or unreachable =:url= argument,
   - non-existent local files passed as =:url=,
   - non-zero ROBOT exit, with stderr lost,
   - empty result sets (the ob-sparql table converter throws),
   - non-ASCII characters in queries.

3. *Global side effect.*  =advice-add= is installed at top level,
   so the ELOT-specific behaviour is active in every Emacs session
   that has loaded ELOT, even in non-ELOT buffers.  The advice body
   does guard on =(elot--is-elot-buffer)=, but MELPA still
   discourages global advice installed at =require= time.  This
   item overlaps with [[#milestone-3-side-effects][Milestone 3]] Step 3.3 — the advice-move work
   is done *here* (Step 1.5) and Milestone 3 only needs to confirm
   it.

The relevant code is concentrated in:
- =elot-package/elot.el= (≈ lines 460–546): prefix-block builder
  =elot-prefix-block-from-alist= and the advice
  =elot--custom-org-babel-execute-sparql=.
- =elot-package/elot-label-display.el= (≈ lines 170–250): label
  retrieval helpers that build SPARQL queries directly.
- =elot-package/elot-mode.el= (≈ lines 280–300): tempo templates
  =elot-block-sparql-select= and =elot-block-sparql-construct=.

** DONE Step 1.2 — De-duplicate prefix declarations
CLOSED: [2026-05-06 ons. 23:15]
- *File*: =elot-package/elot.el=.
- Context: =elot--custom-org-babel-execute-sparql= already
  unconditionally prepends the prefix block built from
  =org-link-abbrev-alist-local= to the query body, regardless of
  whether the body contains its own inline =PREFIX= lines.  The
  earlier note in this plan (claiming the prefix block was only
  prepended when no inline =PREFIX= lines were present) was
  incorrect — confirmed by reading =elot.el= ≈ lines 522–530.
  Concatenation today therefore yields *duplicate* =PREFIX=
  declarations when the body already declares the same label,
  which most SPARQL engines tolerate but ROBOT and some endpoints
  reject (or warn about) when the URIs disagree.
- Change the prefix handling so the merged query has *exactly one*
  declaration per prefix label, with the following semantics:
  1. Build the merged sequence of declarations as
     =(injected-from-alist ++ inline-from-body)=, preserving
     source order within each part.  "Inline" means a top-of-body
     =PREFIX foo: <uri>= line as recognised by SPARQL 1.1 syntax
     (case-insensitive keyword, optional surrounding whitespace).
  2. Walk the merged sequence and collapse by prefix label such
     that *the last declaration wins*.  Because the alist-derived
     declarations come first, this means an inline declaration in
     the user's body always overrides one injected from
     =org-link-abbrev-alist-local= — i.e. the user can override
     ELOT's project-wide prefix mapping by writing =PREFIX foo:
     <other-uri>= at the top of the block.
  3. If two declarations share a label but disagree on the URI,
     emit a single =display-warning= (category =elot-sparql=,
     level =:warning=) naming the label, both URIs, and which
     one wins ("last").  Identical re-declarations (same label
     and same URI) are silently de-duplicated.
  4. Strip the now-redundant inline =PREFIX= lines from the body
     before re-assembling the query, so the final string contains
     each prefix exactly once at the top.
- Factor the merge/dedup logic into a pure helper, e.g.
  =elot--sparql-merge-prefixes (alist body) → (prefix-block
  . body-without-prefix-lines)=, so it can be unit-tested without
  invoking ROBOT or any endpoint.
- Goal (unchanged): a query block containing only the
  user-written =SELECT= / =CONSTRUCT= body — no =PREFIX=
  boilerplate, no noweb call — executes correctly.  *Additional*
  goal from this step: a query block that *does* carry inline
  =PREFIX= lines (legacy or override) executes without producing
  duplicate declarations, and conflicting overrides are surfaced
  as warnings rather than silently shadowed.
- Add ERT tests under =test/= for the helper covering: empty
  body; body with no inline prefixes; body with inline prefixes
  fully covered by the alist (identical URIs → silent dedup);
  body with inline prefixes that override the alist (different
  URIs → warning, last wins); body with inline-only prefixes not
  in the alist (preserved verbatim).  These tests do not require
  ROBOT.  A separate end-to-end test that actually executes a
  query through the advice should be gated with =skip-unless=
  on ROBOT availability.

** DONE Step 1.4 — Harden =elot--custom-org-babel-execute-sparql=
- *File*: =elot-package/elot.el= (mirrored in =elot-defs.org=
  blocks =src-robot-query= and =src-sparql-exec-patch=; see
  Step 1.7 for the formal tangle round-trip check).

The advice body and =elot-robot-execute-query= were factored into
named helpers so each failure mode is exercised by a unit test
without standing up ROBOT.  The new helpers, all in =elot.el=:

| Helper                              | Responsibility                                                                                              |
|-------------------------------------+-------------------------------------------------------------------------------------------------------------|
| =elot--sparql-classify-url=         | Validates =:url=.  Returns the symbol =endpoint= or =local-file=, or signals a =user-error=.                |
| =elot--sparql-resolve-format=       | Normalises =:format= to the symbol =csv= or =ttl=.  Warns (=display-warning= category =elot-sparql=) on unrecognised non-empty values; nil/empty silently default to =csv=. |
| =elot--sparql-result-empty-p=       | Detects an empty result buffer (totally empty, whitespace-only, or — for =csv= — header-row-only).          |
| =elot--sparql-empty-result-sentinel= | Constant string returned in place of an empty CSV body so =org-babel-sparql-convert-to-table= is not invoked on input it cannot parse. |

Failure modes addressed (one-to-one with the original checklist):

1. *=:url= not provided.*  =elot--sparql-classify-url= signals a
   =user-error= naming the missing parameter and explaining what
   to set it to.  Also catches non-string =:url= values.
2. *Non-existent local file.*  The same helper resolves =:url=
   via =expand-file-name= and signals a =user-error= that names
   both the resolved absolute path and the current
   =default-directory=.  =elot-robot-execute-query= performs the
   same check defensively as a second line of defence, and
   additionally errors out if =elot-robot-jar-path= is unset or
   missing.
3. *ROBOT non-zero exit.*  =elot-robot-execute-query= no longer
   shells out via =elot-robot-command= (which uses
   =shell-command= and discards stderr).  It now uses
   =call-process= directly with stderr captured to a temp file,
   and on non-zero exit raises =error= whose message includes the
   exit code and ROBOT's trimmed stderr.  The temp file is
   cleaned up via =unwind-protect=.
4. *Empty result set.*  The advice now consults
   =elot--sparql-result-empty-p= before calling
   =org-babel-sparql-convert-to-table=.  An empty CSV result
   yields =elot--sparql-empty-result-sentinel= ("[empty result
   set]") instead of a backtrace.
5. *Non-ASCII / multibyte content.*  Both the advice body and
   =elot-robot-execute-query= bind =coding-system-for-read= and
   =coding-system-for-write= to =utf-8= around their
   =with-temp-buffer= / =with-temp-file= forms.  The Java
   sub-process additionally receives =-Dfile.encoding=UTF-8= so
   its parsing of the query file and writing of the result file

***** Root cause and fix (2026-05-??)

The fault was in =elot--sparql-prefix-line-re= (the regexp used by
=elot--sparql-strip-inline-prefixes=, called via
=elot--sparql-compute-merged-prefixes=), not in the merge logic
itself.  The regexp anchored on =\\'= immediately after optional
whitespace following the closing =>=, so any inline PREFIX line
carrying a trailing SPARQL line-comment failed to match.

Inspection of =test/fixtures/sparql-roundtrip.org= shows that
Blocks 2, 3 and 4 all have trailing =# ...= comments on their
inline PREFIX lines (Block 7 — which passes — does not).  When
the line failed to match:

- =elot--sparql-strip-inline-prefixes= returned an empty inline
  pair list and a body that still contained the inline =PREFIX=
  line verbatim;
- =elot--sparql-compute-merged-prefixes= therefore saw no
  conflict between the alist's =rdfs:= and the body's =rdfs:=,
  and emitted no warning;
- the body — re-prepended with the alist's prefix block —
  still parsed at ROBOT because SPARQL allows a second =PREFIX
  rdfs:= declaration to shadow the first, so domain/range came
  back unbound.  This is exactly the symptom observed for
  Block 3 (correct result, missing warning).

optional =# ...= line-comment between the closing =>= and end
of string:

  "\\`[ \t]*PREFIX[ \t]+\\([^[:space:]:]+\\):[ \t]*<\\([^>]*\\)>[ \t]*\\(?:#.*\\)?\\'"

#+end_src

lines as intended, the conflict-detection branch fires for
Block 3 (and harmlessly for Block 4, where the override is
unused), and the round-trip test goes green.

* Milestone 2 — Resolve legacy =elot.el= LOB pipeline
:PROPERTIES:
:CUSTOM_ID: milestone-2-legacy-cleanup

** Background

=elot-package/elot.el= still contains ~20 functions left over from
the pre-rewrite "Library of Babel" (LOB) pipeline.  They are no
longer used by =elot-tangle.el=, but until Milestone 1 they were
also reachable through:
- =elot-package/elot-lob.org=, whose named blocks
  =resource-declarations= and =resource-taxonomy= called
  =elot-resource-declarations-from-header= and
  =elot-resource-taxonomy-from-header=; and
- example files that invoke those named blocks via
  =<<resource-declarations(...)>>= noweb references.

After Milestone 1 the SPARQL path no longer needs the LOB prefix
block at all.  The remaining LOB calls in example files are
*silently ignored* by the new tangle pipeline: an old file that
contains =<<resource-declarations(...)>>= still exports correctly
to OWL because the new pipeline does not look at those noweb
calls.  This means the LOB functions can be deleted outright
without breaking existing ontology sources.

One legacy function — =elot-prefix-block-from-alist= — is still
called from production code in =elot-label-display.el= (line ≈173)
and (after Milestone 1) from the rewritten SPARQL advice.  It must
be preserved (relocated, not deleted).

** Decision: Strategy B with selective LOB retention

We delete the legacy LOB *pipeline* functions (the
=resource-declarations= / =resource-taxonomy= family) but *retain
=elot-lob.org= as a supported, user-facing part of ELOT* for
=#+call:=-style helpers — most importantly =rdfpuml-block=, which
authors invoke from prose to render TTL diagrams with caption and
results integration.

Rationale:

- After Milestone 1, no production code path depends on the
  legacy pipeline LOB blocks.
- Old example files that still contain =<<resource-declarations(...)>>=
  noweb calls continue to export to OWL correctly via the new
  tangle pipeline (those calls are silently ignored).
- Only one example, =examples/reasoner-example.org=, contains the
  old SPARQL prefix boilerplate.  It is a special-case file and
  is acceptable to leave as-is — it does not gate the milestone.
- =rdfpuml-block= and similar =#+call:=-style helpers are a
  *different category*: they are idiomatic Org (named,
  parameterised, caption-bearing, results-integrated, and they
  consume another named block as input via =org-babel-ref-resolve=).
  No Elisp-only equivalent is as compact for the author.  These
  stay, are documented as ELOT API, and =elot-lob.org= is
  auto-ingested when =elot-mode= enables.
- This avoids carrying ~20 dead pipeline functions plus a
  deprecation schedule into the MELPA submission, while
  preserving an ergonomic authoring affordance users rely on.

*** Sub-step 4.2.1 — Harden the optional =hydra= dependency (Option A)

- *Files*: =elot-package/elot.el=, =elot-defs.org=.
- Problem: =elot.el= currently wraps the hydra menu definition in
  a runtime guard:
  #+begin_src emacs-lisp
  (when (fboundp 'defhydra)
    (defhydra elot-hydra (:color blue :hint nil) ...))
  #+end_src
  =defhydra= is a *macro*, so the byte-compiler expands the body
  whenever hydra happens to be on =load-path= at compile time.
  The expansion bakes in references to hydra-internal functions
  (e.g. =hydra--call-interactively-remap-maybe=).  If the
  resulting =.elc= is then loaded in an Emacs *without* hydra on
  =load-path=, =(require 'elot)= fails — the runtime
  =(when (fboundp 'defhydra) ...)= guard does not protect against
  references injected by macro-expansion.
- This is exactly what surfaced when the new
  =sparql-advice-test= was added: the test's batch Emacs picks
  up hydra (probably via a leaking user package directory or a
  transitive dep), the helper module gets compiled with
  hydra-aware expansion, and a load in a clean environment
  blows up.
- *Fix (Option A): defer macro expansion to runtime.*  Replace
  the =(when (fboundp 'defhydra) (defhydra elot-hydra ...))=
  form with =(with-eval-after-load 'hydra (eval '(defhydra
  elot-hydra ...) t))=.  Concretely:

  #+begin_src emacs-lisp
  ;; The defhydra macro reference must NOT be expanded at
  ;; byte-compile time, because hydra is an optional dependency
  ;; (see Package-Requires).  Defer expansion until hydra is
  ;; actually loaded at runtime.
  (with-eval-after-load 'hydra
    (eval
     '(defhydra elot-hydra (:color blue :hint nil)
        "..."
        ...)
     t))
  #+end_src

  Notes on the construction:
  - =with-eval-after-load 'hydra= ensures the form runs only
    once hydra is genuinely available.
  - =(eval 'FORM t)= (lexical) prevents the byte-compiler from
    looking inside =FORM= and demanding =defhydra= at compile
    time.  Without the quoted-and-=eval='d wrapper the compiler
    would still try to expand the macro because the surrounding
    =with-eval-after-load= form is itself byte-compiled.
- The S-F5 keybinding in =elot-mode.el= already guards on
  =(fboundp 'elot-hydra/body)= (line ≈500), so the keymap
  installation continues to work whether or not hydra ever
  loads.  No change needed there.
- =elot-setup-org-keybindings= in =elot.el= (≈ line 1257)
  unconditionally calls =(local-set-key elot-key-open-hydra
  #'elot-hydra/body)=.  Wrap that single call in
  =(when (fboundp 'elot-hydra/body) ...)= for symmetry — without
  hydra loaded, =elot-hydra/body= is not defined and the
  keybinding would otherwise resolve to nothing useful.
- Add =(declare-function elot-hydra/body "ext:hydra")= near the
  top of =elot.el= (alongside the other forward declarations) to
  silence the byte-compile warning that would otherwise appear
  for the unconditional reference inside
  =elot-setup-org-keybindings=.
- Mirror the change in =elot-defs.org= (named block
  =src-hydra-menu=) so a re-tangle is a no-op.

* Working with this plan in =gptel=

Practical tips for executing this plan with =gptel=:

- *Narrow before sending.*  Place point on a milestone heading and
  =C-x n s= before invoking =gptel-send=; this scopes the prompt
  to the milestone only.
- *Mark steps as you go.*  Each =TODO Step N.M= heading can be
  cycled to =DONE= with =C-c C-t=.  The cumulative count of DONE
  vs TODO under a milestone is your progress indicator
  (=org-columns= can render this as a percentage).
- *Capture command output back into the document.*  After the
  acceptance check for a milestone, dump the relevant =sh= output
  into a =#+begin_example= block under that heading.  This makes
  the file the durable record of "milestone N was green at
  commit SHA X".
- *Commit per step, not per milestone.*  Even though stability is
  defined per milestone, the working tree should be committable
  after each =TODO Step= so a botched LLM edit can be reset
  without losing earlier good work.
- *Refresh the baseline if you stop and resume.*  If many days
  pass between milestones, redo Step B.1 / B.2 to make sure the
  warning/test baseline still reflects =main=.
