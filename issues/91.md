## Background

Currently, ELOT assumes a single ontology per Org file. This is enforced when collecting resource labels (label-display) and when tangling/exporting OMN sections. Only the first occurrence of an ontology context is considered. However, there are strong use cases for having several distinct ontologies within the same Org file, especially for workflows involving interconnected small ontologies or testing imports.

Typical structure for a multi-ontology ELOT file:
```
* Ontology A
:PROPERTIES:
:ID: ont-a
:ELOT-context-type: ontology
:ELOT-context-localname: ont-a
:ELOT-default-prefix: a
:header-args:omn: :tangle ./ont-a.omn :noweb yes
:END:
...
* Ontology B
:PROPERTIES:
:ELOT-context-type: ontology
:ELOT-context-localname: ont-b
:ELOT-default-prefix: b
:header-args:omn: :tangle ./ont-b.omn :noweb yes
:END:
...
```

## Desired improvements

- [ ] Update `elot-slurp-entities` to collect labels from **all** ontology contexts in the file, not just the first.
    - Loop over all `:ELOT-context-type: ontology` headings and gather resource sections and labels for each.
- [ ] Update `elot-update-link-abbrev` to merge prefix tables from all ontology contexts into a single `org-link-abbrev-alist-local` (the union), with the following rules:
    - **Same prefix key, same URI:** merge silently — this is normal.
    - **Different prefix keys, same URI:** merge, but issue a **warning** — this is bad practice (multiple prefixes pointing to the same namespace).
    - **Same prefix key, different URIs:** this is an **error** — issue a serious warning; first occurrence (in file order) wins and subsequent conflicting entries are ignored. The user must resolve this manually.
    - **Disjoint prefixes:** merge silently.
- [ ] When the same URI is declared under multiple ontology headings with different labels, issue a warning and use the first occurrence for label-display overlays.
- [ ] Warnings should be emitted at file-open time and again on save when `elot-update-link-abbrev` re-runs.
- [ ] Ensure tangling and OMN export are unaffected — each ontology's `:header-args:omn:` already handles per-ontology tangle targets correctly.
- [ ] Document best practices: how to structure multi-ontology files, how to avoid prefix conflicts, and the "first occurrence wins" policy for both label and prefix conflicts.

## Acceptance criteria

- [ ] All per-ontology prefixes are merged into `org-link-abbrev-alist-local`; resources from all ontologies are available as abbreviations in label-display and export.
- [ ] Label-display overlays cover resources from all ontologies in the file.
- [ ] Same prefix key, different URI: serious warning emitted; first occurrence used.
- [ ] Different prefix keys, same URI: warning emitted (bad practice); both keys available.
- [ ] Same prefix key, same URI in multiple ontologies: merged silently.
- [ ] URI label conflicts across ontologies: warning emitted; first occurrence used for overlays.
- [ ] No regressions for single-ontology files.
- [ ] Documentation updated with examples and edge case descriptions.

---

**Rationale:**
This change would allow users to experiment, cross-reference, and develop sets of related ontologies in one Org file without being forced to split files or rework label-display overlays. It also enables robust multi-ontology workflows for modular domain modeling, ontology imports, and testing interoperability.
