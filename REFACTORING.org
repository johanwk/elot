* Data Flow

The pipeline converts Org structure into OMN output.

The entry point is =elot-tangle-buffer-to-omn= (called directly or
via =org-babel-tangle= through =elot-tangle-around-advice=).  The
pipeline runs in roughly nine steps:

1. *Sanity checks* — =elot-sanity-check-prefixes= and
   =elot-sanity-check-resource-sections= silently add any missing
   property-drawer entries (=:prefixdefs: yes=, =:resourcedefs: yes=,
   =:ID:=) so the later traversal can rely on them being present.

2. *Parse buffer* — =org-element-parse-buffer= reads the Org buffer
   into a pure AST (no buffer motion).

3. *Build hierarchy* — =elot-parse-headline-hierarchy= walks the AST
   with a stack-based traversal and builds =elot-headline-hierarchy=:
   a nested plist tree whose nodes carry =:title=, =:level=, =:uri=
   (obtained by =elot-entity-from-header=), =:label=, =:descriptions=
   (description-list key/value pairs), =:children=,
   =:tangle-target-omn=, =:prefixes=, =:prefixdefs=, =:resourcedefs=,
   and positional markers.

4. *Update prefix abbreviations* — =elot-update-link-abbrev= scans the
   hierarchy for nodes with =:prefixdefs "yes"= and populates
   =org-link-abbrev-alist-local= so that CURIE expansion works in later
   annotation-value formatting.

5. *Generate prefix block* — for each top-level ontology node,
   =elot-omn-prefix-block= locates the prefix sub-node and formats the
   =Prefix:= declarations that open the OMN file.

6. *Generate resource frames* — =elot-omn-resource-declarations=
   recursively visits all children of the ontology node.  For each
   resource it calls =elot-omn-resource-frame=, which emits the OMN
   frame keyword (=Class:=, =ObjectProperty:=, etc.) followed by
   annotations (=elot-omn-format-annotations=) and restrictions such as
   =SubClassOf:= (=elot-omn-format-restrictions=).

7. *Merge with user OMN blocks* — =elot-tangle-around-advice= also
   invokes the normal =org-babel-tangle= pass, which extracts any
   literal =#+begin_src omn= source blocks.  Their content is appended
   after the outline-generated OMN in the target file.

8. *Write output file* — the combined OMN string is written to the path
   declared in =:HEADER-ARGS\:OMN: :tangle <file>.omn=.  Any Emacs
   buffer already visiting that file is silently reverted.

9. *Post-process to Turtle* — =elot-robot-omn-to-ttl= invokes ROBOT
   (=java -jar robot.jar convert=) to validate and convert the =.omn=
   file into a =.ttl= Turtle file.  In interactive mode this runs
   asynchronously; in batch mode it runs synchronously and calls
   =kill-emacs= on failure.

* Design Decisions

This section captures design notes and rationale as they arise during
the work.

** Strategy B for the legacy Library-of-Babel pipeline (2026-05-08)

The pre-rewrite ELOT pipeline emitted OMN via a family of
Library-of-Babel named blocks in =elot-package/elot-lob.org=
(=resource-declarations=, =resource-taxonomy=) that delegated to
roughly twenty helper defuns in =elot.el= (=elot-omn-declare=,
=elot-resource-declarations-from-header=,
=elot-resource-taxonomy-from-header=, the description-list and
tag-parsing helpers, etc.).  The new tangle pipeline documented
under "Data Flow" above replaces all of this.

Two strategies were considered for the dead code:

- *Strategy A* - keep the legacy defuns and LOB blocks around,
  marked as deprecated, with a removal schedule.
- *Strategy B* - delete the legacy defuns and the two retired LOB
  blocks outright.

*Decision: Strategy B*, taken as part of the MELPA submission
preparation (see =ELPA-SUBMISSION-PLAN.org= Milestone 2).

Rationale:

1. The SPARQL rewrite (Milestone 1) removed the last production
   dependency on =elot-prefix-block-from-alist='s LOB role; the
   function itself was relocated to =elot-tangle.el= and is
   still called from =elot-label-display.el= and the rewritten
   SPARQL advice.
2. A grep over =elot-package/= and =examples/*.org= confirmed
   that the legacy defuns and the two retired LOB blocks
   (=resource-declarations=, =resource-taxonomy=) had no
   remaining call sites.  Their only "callers" were each other.
3. Old example files that still contain
   =<<resource-declarations(...)>>= noweb references continue to
   tangle correctly to OMN: the new pipeline does not consult
   those noweb calls and silently ignores them.  No user file
   needs editing.
4. The single example with hand-written SPARQL =PREFIX=
   boilerplate (=examples/reasoner-example.org=) is treated as a
   special case and left untouched - it does not gate the
   milestone.
5. =elot-lob.org= itself is *retained* selectively for
   =#+call:=-style helpers (=rdfpuml-block=, =kill-prefixes=,
   theme/setupfile helpers, =robot-metrics=, etc.).  These are
   idiomatic Org affordances that have no compact Elisp
   equivalent and remain part of ELOT's supported user-facing
   API.  =elot-mode= now auto-ingests =elot-lob.org= on enable so
   users no longer need to run =M-x org-babel-lob-ingest=
   manually.

Carrying ~20 dead defuns plus a deprecation schedule into the
MELPA submission was judged worse than a clean break, given that
no in-tree or known downstream caller exercises the removed
surface.
